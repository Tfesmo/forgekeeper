# Streaming & Telemetry Design

## Overview

This document covers two real-time streaming features for Forgekeeper:

1. **LLM Response Streaming** — SSE endpoint that streams LLM chunks as they are generated by llama.cpp.
2. **Telemetry Monitoring** — SSE endpoint that streams extracted log metrics from monitored files using `@logdna/tail-file`.

---

## 1. LLM Response Streaming

### Route

```
GET /api/stream/llm?session=<sessionId>&prompt=<prompt>
```

### Flow

1. Client connects via SSE to `/api/stream/llm`.
2. Server receives `session` and `prompt` query params.
3. `callLLM()` is invoked with `stream: true`, streaming chunks to the SSE response.
4. Each chunk is written as an SSE event: `data: <chunk>\n\n`.
5. On client disconnect (`req.on('close')`), the LLM call is aborted via signal/abort controller.

### Integration with `llmService.js`

- `callLLM()` already accepts `stream` param and yields tokens one by one via async generator.
- The SSE endpoint wraps the generator, writing each yielded token as an SSE message.

```js
// In chatRoutes.js
router.get('/api/stream/llm', async (req, res) => {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    Connection: 'keep-alive',
  });

  const { session, prompt } = req.query;
  const abortController = new AbortController();

  req.on('close', () => abortController.abort());

  try {
    const stream = callLLM({ prompt, stream: true, sessionId: session }, abortController.signal);
    for await (const chunk of stream) {
      if (abortController.signal.aborted) break;
      res.write(`data: ${JSON.stringify({ type: 'chunk', content: chunk })}\n\n`);
    }
    res.write(`data: ${JSON.stringify({ type: 'done' })}\n\n`);
    res.end();
  } catch (err) {
    res.write(`data: ${JSON.stringify({ type: 'error', message: err.message })}\n\n`);
    res.end();
  }
});
```

### Constraints

- Memory: Each SSE message is written immediately — no buffering.
- Abort: `AbortController` signal passed to `callLLM` to cancel generation on client disconnect.
- No retry logic — SSE reconnection handled by client (`EventSource`).

## 2. Telemetry Monitoring

### Route

```
GET /api/stream/telemetry?file=<path>&interval=<ms>
```

### Flow

```
Monitored File → @logdna/tail-file → Extractor → Aggregator → SSE Client(s)
```

### 2.1 File Monitoring with `@logdna/tail-file`

`@logdna/tail-file` is a lightweight, zero-dependency file tail module.

- Opens file in append mode, reads new lines incrementally.
- Returns an async iterable of new lines.
- Supports backpressure via async iteration.
- Tracks file position internally (no manual offset management).
- Single file only (no rotation support).

```js
import { Tail } from '@logdna/tail-file';

const tail = new Tail('/path/to/logfile.log', {
  interval: 500,   // poll interval (ms) — default 500
  seek: true,      // read from end on startup
});

for await (const line of tail) {
  // process new line
}
```

### 2.2 Config-Driven Extractor

Extractors are defined in `telemetry.json` — users specify regex patterns and field mappings. This makes the system flexible: users can add new log formats without code changes.

```json
// src/config/telemetry.json
{
  "extractors": [
    {
      "name": "error_extractor",
      "file": "/var/log/app.log",
      "pattern": "ERROR\\s+(?<timestamp>\\d{4}-\\d{2}-\\d{2}T[\\d:]+)\\s+(?<message>.+)",
      "fields": ["timestamp", "message"]
    },
    {
      "name": "request_extractor",
      "file": "/var/log/app.log",
      "pattern": "(?<method>GET|POST|PUT|DELETE)\\s+(?<path>\\S+)\\s+\\(?<status>\\d{3}\\)",
      "fields": ["method", "path", "status"]
    }
  ]
}
```

**Extractor logic:**

1. Load extractors from `telemetry.json`.
2. For each new line from the tail stream, try matching each extractor's regex.
3. If matched, extract named groups into a structured object.
4. Pass the extracted object to the aggregator.

```js
function createExtractor(extractorDef) {
  const regex = new RegExp(extractorDef.pattern);
  return function extract(line) {
    const match = line.match(regex);
    if (!match) return null;
    const fields = {};
    for (const field of extractorDef.fields) {
      fields[field] = match.groups[field];
    }
    return fields;
  };
}
```

### 2.3 Aggregator

The aggregator collects extracted objects and computes metrics over different window types:

| Window       | Description                              | Example                          |
|-------------|------------------------------------------|----------------------------------|
| Session      | All data since the SSE connection started | Total errors since client connected |
| Rolling      | Fixed-size time window (default 5 min)   | Error rate in last 5 minutes     |
| Prompt       | Extracted data per user prompt           | Tokens generated per prompt      |

**Aggregator structure:**

```js
class Aggregator {
  constructor(windows = ['session', 'rolling', 'prompt']) {
    this.windows = windows;
    this.sessionData = [];
    this.rollingWindowMs = 5 * 60 * 1000; // 5 min default
    this.rollingData = [];
    this.promptData = new Map(); // promptId -> aggregated data
  }

  add(extracted) {
    this.sessionData.push({ ...extracted, timestamp: Date.now() });
    this.rollingData.push({ ...extracted, timestamp: Date.now() });

    // Prune rolling window
    const cutoff = Date.now() - this.rollingWindowMs;
    this.rollingData = this.rollingData.filter(d => d.timestamp >= cutoff);
  }

  getSnapshot() {
    return {
      session: this.aggregateSession(),
      rolling: this.aggregateRolling(),
    };
  }

  aggregateSession() {
    return {
      total: this.sessionData.length,
      errors: this.sessionData.filter(d => d.type === 'error').length,
    };
  }

  aggregateRolling() {
    return {
      total: this.rollingData.length,
      errors: this.rollingData.filter(d => d.type === 'error').length,
    };
  }
}
```

### 2.4 Telemetry SSE Endpoint

```js
// In telemetryRoutes.js
import { Tail } from '@logdna/tail-file';
import { readFileSync } from 'fs';
import { resolve } from 'path';

function loadExtractors() {
  const config = JSON.parse(readFileSync(resolve(__dirname, '../config/telemetry.json'), 'utf-8'));
  return config.extractors.map(def => createExtractor(def));
}

router.get('/api/stream/telemetry', (req, res) => {
  const { file, interval = 500 } = req.query;

  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    Connection: 'keep-alive',
  });

  const extractors = loadExtractors();
  const aggregator = new Aggregator();

  // Find matching extractor for the requested file
  const fileExtractor = extractors.find(e => e.file === file);
  if (!fileExtractor) {
    res.write(`data: ${JSON.stringify({ type: 'error', message: 'No extractor configured for file' })}\n\n`);
    res.end();
    return;
  }

  const tail = new Tail(file, { interval: parseInt(interval) });

  (async () => {
    try {
      for await (const line of tail) {
        const extracted = fileExtractor(line);
        if (extracted) aggregator.add(extracted);

        const snapshot = aggregator.getSnapshot();
        res.write(`data: ${JSON.stringify({ type: 'snapshot', ...snapshot })}\n\n`);
      }
    } catch (err) {
      res.write(`data: ${JSON.stringify({ type: 'error', message: err.message })}\n\n`);
    }
  })();

  req.on('close', () => {
    tail.stop();
    res.end();
  });
});
```

### 2.5 Telemetry Config (`telemetry.json`)

```json
// src/config/telemetry.json
{
  "extractors": [
    {
      "name": "error_extractor",
      "file": "/var/log/forgekeeper.log",
      "pattern": "ERROR\\s+(?<timestamp>\\d{4}-\\d{2}-\\d{2}T[\\d:]+)\\s+(?<message>.+)",
      "fields": ["timestamp", "message"]
    },
    {
      "name": "request_extractor",
      "file": "/var/log/forgekeeper.log",
      "pattern": "(?<method>GET|POST|PUT|DELETE)\\s+(?<path>\\S+)\\s+(?<status>\\d{3})",
      "fields": ["method", "path", "status"]
    }
  ],
  "metrics": {
    "aggregationWindows": ["session", "rolling", "prompt"],
    "rollingWindowMs": 300000,
    "pollIntervalMs": 500
  }
}
```

Users can:

- Add new extractors by adding entries to the `extractors` array.
- Modify regex patterns for different log formats.
- Define which fields to extract via the `fields` array.
- Configure aggregation window types and sizes.

### 2.6 Memory Budget

| Component    | Estimated Memory | Notes                              |
|-------------|-----------------|------------------------------------|
| Monitor     | ~16 KB          | Tail file handle + position buffer |
| Extractor   | ~12 KB          | Regex objects + compiled patterns  |
| Aggregator  | ~20 KB          | Session + rolling arrays           |
| **Total**   | **~54 KB**      | Well under 1 MB budget             |

- Rolling window prunes entries older than `rollingWindowMs`.
- Session data is bounded by SSE connection lifetime.
- No accumulation across clients — each client gets its own aggregator instance.

## 5. Critique & Considerations

### LLM Streaming
- **Pro**: Leverages existing async generator in `callLLM` — minimal new code.
- **Con**: No retry/reconnect — client must reconnect manually via `EventSource`.
- **Note**: If `callLLM` doesn't accept an abort signal, we need to add one (e.g., via `AbortController`).

### Telemetry
- **Pro**: `@logdna/tail-file` is zero-dependency, lightweight, and stream-based.
- **Pro**: Config-driven extractors allow flexible log parsing without code changes.
- **Con**: No log rotation support — acceptable per constraints (single file monitoring).
- **Con**: No backpressure between SSE writes and tail reads — if client is slow, SSE buffer could grow. Mitigation: throttle emission if `res.write()` returns false.
- **Note**: Each SSE client gets a separate tail instance — for high concurrency, consider a shared tail with multiple aggregators.
